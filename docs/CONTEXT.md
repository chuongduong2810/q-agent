# Q-Agent — Working Context & Glossary

Living glossary for the Q-Agent build. Terms here are the **shared vocabulary**
used across the backend, frontend, and docs. Implementation detail lives in code
and the ADRs — this file only pins down *meaning*.

## Product in one line

A **local-first, AI-powered QA Operating System**: sync tickets from Azure
DevOps / Jira → analyze requirements with Claude CLI → generate manual test
cases → human review → generate Playwright automation → execute → collect
evidence → publish results back to the originating ticket.

## Core entities

| Term | Meaning |
|------|---------|
| **Provider** | An external system Q-Agent talks to: Azure DevOps, Jira, GitHub. Responsible for syncing tickets, listing a project's repositories, publishing comments, updating status, uploading attachments. Credentials are stored encrypted at rest. |
| **Project** | A connected project from a provider (e.g. an ADO project or Jira project key). Organizes tickets, and owns one or more **Repositories**. |
| **Repository** | A git repo belonging to a project (a project usually has several). Discovered from the provider (ADO/GitHub) or added manually. One repo is flagged **default** — the app it targets when a run doesn't specify one. Each repo has its own **Project Knowledge Base**. |
| **Project Config** | User-authored, project-level runtime settings edited on the Project Details → Settings page: base URL, per-environment URLs, **test accounts** (passwords encrypted at rest, masked in the UI), the repository list, and arbitrary extra key/values. Feeds downstream generation so specs run with little to no manual edits. |
| **Project Knowledge Base** | What `project-bootstrap` learns about a **repository** (stack, architecture, domain, routes, real selectors, auth/login flow, environments, reusable Page Objects/fixtures) — persisted per repo as `knowledge.md` + `knowledge.json`. Built once per repo (rebuilt on material change) and reused by every downstream AI action instead of re-reading the code. May be **enriched at runtime** by the Exploration Agent with `verified_at_runtime` entries. |
| **Exploration Agent** | An optional, user-triggered agent that drives a **real browser** over the running app in an observe→decide→act (ReAct) loop to discover the real routes/selectors for a target screen, then writes them back into that repository's Knowledge Base. It does **not** generate tests — it enriches the KB so the deterministic `automation-generator` can. Bounded by mandatory **stop conditions** (see below). Scoped to one repository. See [ADR 0010](adr/0010-dom-exploration-agent-kb-enrichment.md). |
| **Verified-at-runtime** | A KB route/selector entry that was observed on the live app by the Exploration Agent (or a passing DOM-grounded self-heal), stamped with a timestamp and the locator **strategy** that worked (`data-testid` → role → label). Preferred over source-inferred entries during generation, and never overwritten by a later merge. |
| **Exploration session** | One bounded run of the Exploration Agent against a target (a ticket/screen + repository). Halts on the first **stop condition**: `done` (goal covered), **step cap** (max actions/session), **repeat detection** (revisits an already-seen state), or **cost budget** (per-session Claude spend ceiling). Produces discovered KB additions + an ordered **exploration log**. |
| **Ticket** | A read-only work item imported from a provider (User Story / Task / Bug / Feature). Carries description, acceptance criteria, comments, attachments, labels, priority, status, linked PRs. |
| **Run** | **The central entity.** One QA session over one or many tickets (single / selected / assigned / sprint). Everything after ticket selection belongs to a Run. Moves through the pipeline stages below. |
| **Test Case** | An ADO-style manual test case generated by Claude for a ticket: title, precondition, steps (action → expected), priority, test type, automation type. The automation type (`Playwright` / `Selenium` / `Cypress` / `Manual`) is editable in the Review Center; only non-`Manual` cases are automatable. |
| **Approval** | A reviewer's decision on a test case: `pending` → `approved` / `rejected`. Only **approved** cases proceed to create-and-link and automation. |
| **Create & Link** | Creating the approved test cases in the provider and linking them to their work items (the `sync` pipeline stage). Supports a **local (dry-run) mode** that records cases locally only and never writes to the provider — to avoid cluttering a live project during development. |
| **Automation Spec** | A Playwright + TypeScript `*.spec.ts` file generated by Claude from an approved, non-Manual test case. Grounded in the repo's Project Knowledge Base + Project Config, so real base URL, routes, selectors and (per project choice) test-account credentials are baked in rather than left as placeholders. |
| **Execution** | A real Playwright run of the approved suite within a Run. Produces per-case status: `pending` / `running` / `pass` / `fail`. |
| **Evidence** | Artifacts captured per executed case: screenshots, video, Playwright trace, console logs, network logs, summary. Grouped by ticket. |
| **Annotation** | Markup drawn on a screenshot (rectangle / arrow / highlight / circle / text) via Pillow, attached to a ticket comment. |
| **Report** | Aggregated run outcome: overall result, per-ticket summary, pass/fail counts, AI failure analysis, timing, environment, evidence links. |
| **Comment / Publish** | The prepared result comment pushed back to the provider ticket, with configurable status mapping (e.g. Ready for QA → Testing → Passed / QA Failed). |
| **Locale / Language** | The frontend display language: **English (`en`, source)** or **Vietnamese (`vi`)**. A pure client-side display preference persisted in `localStorage` (`qagent.lang`) — orthogonal to routing and never stored on the server. Switching is instant. Does **not** affect data, provider payloads, or generated artifacts (those follow the source ticket's language). See [ADR 0011](adr/0011-frontend-internationalization-en-vi.md). |
| **Translation namespace** | A per-feature bucket of UI strings (`common`, `nav`, `auth`, `settings`, `dashboard`, `projects`, `tickets`, `runs`, `pipeline`, `reports`, `commands`), stored as one JSON file per language under `app/src/i18n/locales/{en,vi}/<ns>.json` and consumed via `useTranslation('<ns>')`. Catalogs are auto-loaded by `import.meta.glob`, so adding a namespace needs no central-config edit. Untranslated keys fall back to English. |
| **Workspace scope** | The per-owner slice of the on-disk workspace. Resolves to `workspace/users/<owner_id>/…` for an owned artifact and `workspace/shared/…` when there is no owner (`owner_id IS NULL`). Each scope holds its own `{specs, evidence, knowledge, repos, auth}` trees, so two users' same-named projects never collide. See [ADR 0009](adr/0009-per-user-workspace-filesystem-and-cloning.md). |
| **Shared namespace** | The admin-managed shared workspace (`owner_id NULL` rows + `workspace/shared/…`). Admins build and maintain reference projects + knowledge there; members **clone from** it but cannot write to it. The filesystem analogue of ADR 0008's shared Claude credential. |
| **Clone (project / knowledge)** | Copying a ready-built project — its `Project` + `ProjectConfig` (incl. encrypted test-account secrets) + `ProjectKnowledge` rows **and** the on-disk `knowledge/`, `repos/`, and `auth/` artifacts — from the shared namespace into the caller's own scope (re-stamping `owner_id`), so the expensive ~20-min `project-bootstrap` build is reused rather than repeated. |

## Pipeline stages (a Run's lifecycle)

`processing` (AI analysis + test-case generation) → `review` → `sync`
(**Create & Link** approved cases to the provider, or record them locally) →
`automation` (spec generation) → `executing` → `evidence` → `comment` (publish)
→ `done`.

This pipeline is the backbone of the UI (a visual **PipelineRail** appears on
every run-scoped screen) and of the backend `Run.status` state machine.

> **Project setup (once per project, before runs):** connect a provider, then on
> **Project Details** add the project's repositories and configure its base URL /
> test accounts, and build a **Project Knowledge Base per repository** with
> `project-bootstrap`. Downstream AI actions read that context.

## Resolved scope decisions (see ADR 0001)

- **Full-stack, local-first.** React frontend + FastAPI backend + SQLite. No cloud.
- **Live integrations, real engines only.** ADO / Jira / GitHub adapters call the
  real REST APIs. Claude CLI is invoked for real (analysis, test-case & spec
  generation). Playwright really executes specs. **No mock/simulation fallback.**
- **Monorepo layout:** `app/` (frontend), `api/` (backend), `scripts/` (setup/start),
  `docs/`, `design/` (approved UI source), `template/` (test-case export templates).
- **UI source of truth:** `design/Q-Agent app design/Q-Agent.dc.html` — the frontend
  reproduces it faithfully, then wires it to the live backend.

## Dedicated AI skills

Every Claude CLI action is guided by a dedicated skill under `skills/<name>/SKILL.md`
(methodology + quality rules + output templates). The backend injects the matching
skill's `SKILL.md` as the Claude **system prompt** for that action (via
`app/services/skills.py`), while the caller's prompt still pins the exact JSON/text
shape the backend parses. Configurable via `QAGENT_SKILLS_DIR` (default `skills/`).

| Backend action | Skill |
|---|---|
| Per-repo knowledge base build | `project-bootstrap` |
| Ticket requirement analysis | `requirement-analyst` |
| Test-case generation / regeneration | `test-case-generator` |
| Playwright spec generation | `automation-generator` |
| AI failure analysis (reports) | `execution-analyzer` |
| Ticket result comment | `ticket-comment-generator` |

`project-bootstrap` runs per repository and, when a local checkout or remote clone
URL is available, **traverses the real source** to discover routes, selectors and
auth. `requirement-analyst`, `test-case-generator` and `automation-generator` then
**consume that Project Knowledge Base + the Project Config** (base URL, routes,
selectors, credentials) so their output reflects the real app.

Other skills ship for future/adjacent use: `test-case-reviewer`,
`automation-reviewer`, `report-generator`, `screenshot-annotator`.

## Personas

- **QA Engineer / QA Lead** (e.g. "Maya Kaur" in the seed data): the primary user.
  Reviews AI work, runs suites, publishes results. Wants an *AI teammate*, not a tool.

## Multi-user (delivered)

Q-Agent is now a **multi-user, logged-in** application with **per-user data
isolation**: authentication (ADR 0007) plus per-user ownership of runs, tickets,
projects, provider connections (each user's own PATs), knowledge, and Claude cost
(ADR 0008). Roles are **admin** / **member** with RBAC enforcement, full member
lifecycle, and self-service. Persistence runs on **PostgreSQL** (SQLite still
supported for local dev) with Alembic migrations. Isolation now extends to the
**workspace filesystem** — per-user artifact trees plus an admin-managed **shared
namespace** members clone from (ADR 0009). See
[`docs/MULTI-USER-MIGRATION-PLAN.md`](MULTI-USER-MIGRATION-PLAN.md) for status.

## Non-goals (current)

CI/CD triggers and Cypress/Selenium runners (architecture stays extensible for
them). **Deferred, blocked on deployment:** an execution job queue/workers and
interactive server-side browser sessions (noVNC → WebRTC). Containerized deployment
(Docker/Compose/TLS) and Tauri desktop packaging remain future work.
