# Q-Agent ÔÇö Working Context & Glossary

Living glossary for the Q-Agent build. Terms here are the **shared vocabulary**
used across the backend, frontend, and docs. Implementation detail lives in code
and the ADRs ÔÇö this file only pins down *meaning*.

## Product in one line

A **local-first, AI-powered QA Operating System**: sync tickets from Azure
DevOps / Jira Ôćĺ analyze requirements with Claude CLI Ôćĺ generate manual test
cases Ôćĺ human review Ôćĺ generate Playwright automation Ôćĺ execute Ôćĺ collect
evidence Ôćĺ publish results back to the originating ticket.

## Core entities

| Term | Meaning |
|------|---------|
| **Provider** | An external system Q-Agent talks to: Azure DevOps, Jira, GitHub. Responsible for syncing tickets, publishing comments, updating status, uploading attachments. Credentials are stored encrypted at rest. |
| **Project** | A connected project from a provider (e.g. an ADO project or Jira project key). Organizes tickets. |
| **Ticket** | A read-only work item imported from a provider (User Story / Task / Bug / Feature). Carries description, acceptance criteria, comments, attachments, labels, priority, status, linked PRs. |
| **Run** | **The central entity.** One QA session over one or many tickets (single / selected / assigned / sprint). Everything after ticket selection belongs to a Run. Moves through the pipeline stages below. |
| **Test Case** | An ADO-style manual test case generated by Claude for a ticket: title, precondition, steps (action Ôćĺ expected), priority, test type, automation type. |
| **Approval** | A reviewer's decision on a test case: `pending` Ôćĺ `approved` / `rejected`. Only **approved** cases proceed to automation. |
| **Automation Spec** | A Playwright + TypeScript `*.spec.ts` file generated by Claude from an approved test case. |
| **Execution** | A real Playwright run of the approved suite within a Run. Produces per-case status: `pending` / `running` / `pass` / `fail`. |
| **Evidence** | Artifacts captured per executed case: screenshots, video, Playwright trace, console logs, network logs, summary. Grouped by ticket. |
| **Annotation** | Markup drawn on a screenshot (rectangle / arrow / highlight / circle / text) via Pillow, attached to a ticket comment. |
| **Report** | Aggregated run outcome: overall result, per-ticket summary, pass/fail counts, AI failure analysis, timing, environment, evidence links. |
| **Comment / Publish** | The prepared result comment pushed back to the provider ticket, with configurable status mapping (e.g. Ready for QA Ôćĺ Testing Ôćĺ Passed / QA Failed). |

## Pipeline stages (a Run's lifecycle)

`processing` (AI analysis + test-case generation) Ôćĺ `review` Ôćĺ `automation`
(spec generation) Ôćĺ `executing` Ôćĺ `evidence` Ôćĺ `comment` (publish) Ôćĺ `done`.

This pipeline is the backbone of the UI (a visual **PipelineRail** appears on
every run-scoped screen) and of the backend `Run.status` state machine.

## Resolved scope decisions (see ADR 0001)

- **Full-stack, local-first.** React frontend + FastAPI backend + SQLite. No cloud.
- **Live integrations, real engines only.** ADO / Jira / GitHub adapters call the
  real REST APIs. Claude CLI is invoked for real (analysis, test-case & spec
  generation). Playwright really executes specs. **No mock/simulation fallback.**
- **Monorepo layout:** `app/` (frontend), `api/` (backend), `scripts/` (setup/start),
  `docs/`, `design/` (approved UI source), `template/` (test-case export templates).
- **UI source of truth:** `design/Q-Agent app design/Q-Agent.dc.html` ÔÇö the frontend
  reproduces it faithfully, then wires it to the live backend.

## Personas

- **QA Engineer / QA Lead** (e.g. "Maya Kaur" in the seed data): the primary user.
  Reviews AI work, runs suites, publishes results. Wants an *AI teammate*, not a tool.

## Non-goals (MVP)

Cloud deployment, multi-user auth/RBAC, CI/CD triggers, Cypress/Selenium runners
(architecture stays extensible for them). Tauri desktop packaging is future.
