1Run-scoped navigation — two structural models

A Run is one QA session over one or more tickets, moving through a fixed pipeline. Screens split into two groups — and run-scoped screens must never be reachable without first choosing a run (which is what lets a reviewer approve the wrong run's cases today).

1aWorkspace mode — sidebar becomes the runRECOMMENDED

Two competing nav surfaces. "Where am I?" is split between sidebar and bar; the pipeline gets a thin strip, so context reads weaker.
Weaker enforcement. Run-scoped items are easy to also surface in the sidebar "for convenience" — reopening the exact silent-default bug.

Recommendation — adopt 1a, Workspace mode

It solves the actual bug structurally: because run-scoped screens exist only after you enter a run, there is no surface from which the app could silently default to "the latest run." The whole shell becoming the run also gives the strongest, always-present context (identity + stage), and the pipeline-as-navigation makes progress obvious — the Linear "project workspace" / Cursor mental model. 1b keeps global permanently visible, but its second nav surface competes for "where am I" and, crucially, invites re-adding run-scoped items to the sidebar — which is how the default-run mistake happens in the first place. The one cost of 1a — global screens are one click behind an exit/mini-row — is fully mitigated by the pinned Global row and ⌘K.
